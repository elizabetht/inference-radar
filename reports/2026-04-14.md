# Inference Radar — 2026-04-14

## Executive Summary

- **SGLang v0.5.10** ships piecewise CUDA graphs as the new default and a GPU staging buffer that consolidates ~30,000 scattered RDMA requests into ~1,000 bulk transfers, delivering ~5x throughput on MoE at concurrency 128+ — no config change required after upgrade.
- **MLPerf Inference v6.0** (April 1): NVIDIA GB300 NVL72 hit 2,494,310 tok/s on DeepSeek-R1 offline — a 2.77x gain over v5.1 driven primarily by Dynamo + TRT-LLM software, not just hardware. DeepSeek-R1, GPT-OSS-120B, and Qwen3-VL-235B are now official benchmark categories.
- **Meta Llama 4** landed open-weight: Scout (17B active / 109B total, 16 experts) and Maverick (17B active / 400B total, 128 experts), both multimodal. Scout fits a single H100 INT4; Maverick on a single DGX host. MoE inference cost drops sharply relative to parameter count.
- **Dynamo KV block leak fix** (#7489): stuck-request rate from 3.1% → 0.018% on client cancel during NIXL KV transfer — a production blocker for PD disaggregated serving is now resolved.
- **TensorRT-LLM** confirms Nemotron-Super-120B NVFP4 on DGX Spark hardware (PR #12830) with official test configs; also fixed a scheduler hang in the PP+disagg+block_reuse+chunked_prefill stack (NVBug 5991576).

## First-Principles Analysis

### 1. SGLang GPU Staging Buffer for PD Disaggregation (`github-sglang-pr-19890`)

**1. Challenge**: Invalidates the assumption that RDMA bandwidth is the PD disaggregation bottleneck. The actual bottleneck was request fragmentation: 30,000 small RDMA transfers per request (one per KV head slice) vs. hardware optimal ~1,000 bulk transfers. Network bandwidth was irrelevant until fragmentation was fixed.

**2. Delete**: Eliminates the case for raw RDMA bandwidth upgrades (faster NICs, more ports) as a PD disagg scaling solution. Also eliminates the workaround of limiting PD concurrency to manage RDMA load.

**3. Simplify**: The minimal form is: pre-allocate a contiguous GPU ring buffer, gather scattered KV slices into it via a single CUDA kernel, then do one RDMA transfer. The GPU-side gather is the key — RDMA hardware prefers large sequential transfers.

**4. Accelerate**: Upgrade spark-01 and spark-02 to SGLang v0.5.10, run current Qwen2.5-7B experiments with `--enable-pd-disagg` at concurrency 32→128. Compare throughput vs v0.5.9 baseline. 1 day runtime. Note: v0.5.10 results are on GB200 (Prefill TP4, Decode DEP4) — DGX Spark's NVLink-C2C is different; the fragmentation fix should generalize but the 5x number likely won't replicate exactly on GB10.

**5. Automate**: Add `sglang-pd-disagg-staging-throughput` to queue.yaml tracking at c=32/64/128 across SGLang versions. This should become a tier-1 recurring benchmark given its direct relevance to the lab's 2-node Spark topology.

---

### 2. Blink: CPU-Free LLM Inference (`arxiv-2604.07609`)

**1. Challenge**: Invalidates the assumption that CPU-side request scheduling is a necessary architectural component of the inference serving stack. The paper demonstrates the CPU adds latency, not value, in the steady-state decode path — 8.47× P99 TTFT reduction comes from eliminating CPU scheduling overhead, not from faster compute.

**2. Delete**: If the CPU bypass principle is valid, it makes the entire design space of "faster async CPU schedulers" (vLLM's V1 scheduler, SGLang's radix tree manager) potentially redundant for throughput-optimized workloads. The correct fix is bypass, not optimization.

**3. Simplify**: The minimal version doesn't require SmartNIC customization. A persistent GPU kernel that manages a ring buffer of pending requests and triggers batch assembly independently of the CPU would capture 80% of the benefit. The SmartNIC RDMA is a bonus.

**4. Accelerate**: DGX Spark has ConnectX-7 NICs. Near-term: measure CPU scheduler contribution to TTFT by profiling vLLM's Python scheduler overhead on spark-01 at c=64/128 using `py-spy`. If >10% of TTFT is in Python scheduling, the CPU hypothesis is confirmed. 2 hours to validate.

**5. Automate**: Not yet — no production framework implements this. File as a watch item. If vLLM or SGLang adopts the persistent GPU batch kernel concept, add to queue immediately.

---

### 3. NIXL EP: MoE Dispatch/Combine Without NVSHMEM (`github-nixl-pr-1341`)

**1. Challenge**: Invalidates the assumption that NVSHMEM is required for high-performance multi-node MoE expert-parallel communication. DeepEP kernels were considered NVSHMEM-dependent; NIXL EP achieves 75–84 GB/s with `nixlPut`/`nixlAtomicAdd` — matching vanilla DeepEP throughput.

**2. Delete**: NVSHMEM container requirements (strict driver/CUDA version pinning) are eliminated as a prerequisite for MoE expert-parallel serving. Removes a significant deployment friction point on DGX Spark where container CUDA versions track NVIDIA's release cadence.

**3. Simplify**: Core mapping: `NVSHMEM nvshmemx_putmem_signal_on_stream` → `nixlPut` + `nixlAtomicAdd`. The kernel is unchanged; only the transport primitive is swapped. ~300 lines of change.

**4. Accelerate**: On spark-01/02 with Dynamo + Qwen3-30B-A3B (MoE model that fits the 2-node setup), compare NIXL EP vs standard NCCL AllToAll for expert dispatch bandwidth and E2E throughput. Setup: 1–2 days (requires Dynamo + NIXL build). Benchmark: 4 hours.

**5. Automate**: Yes — add `dynamo-nixl-ep-moe-throughput` to queue.yaml as a recurring benchmark. The ability to run MoE expert-parallel without NVSHMEM opens up a new class of experiments for the lab.

## Open Questions

1. SGLang's GPU staging buffer shows 5x throughput improvement on GB200 with TP4+DEP4. DGX Spark uses NVLink-C2C (not InfiniBand/RoCE). How much of the gain is RDMA-fragmentation-dependent vs. NVLink-topology-dependent? Is a 2x gain on Spark realistic?

2. With NIXL EP kernels at 75–84 GB/s matching vanilla NVSHMEM/DeepEP, what is the actual bottleneck for MoE expert-parallel on DGX Spark? Is NVLink-C2C (≈900 GB/s aggregate) or CPU dispatch overhead the ceiling for dispatch/combine?

3. Blink's CPU-bypass architecture claims 8.47× P99 TTFT reduction by eliminating host CPU scheduling. Current experiments show TTFT spikes at high concurrency (e.g., 3280ms P99 at c=8 in Exp1 baseline). How much of that P99 spike is CPU scheduler overhead vs. GPU-side batching delay? A `py-spy` profile of the vLLM scheduler under load would answer this in hours.
