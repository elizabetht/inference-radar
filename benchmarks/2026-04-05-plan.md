## Experiment 1 — SGLang Piecewise CUDA Graph Default (HIGH PRIORITY)

**Hypothesis**: v0.5.10's piecewise CUDA graph capture reduces peak GPU memory and improves throughput on MoE models vs v0.5.9's all-or-nothing capture.

**Setup**:
- Node: spark-01 only (single node)
- Model: `mistralai/Mixtral-8x7B-Instruct-v0.1` (fits in 128GB, good MoE benchmark)
- Compare: SGLang v0.5.9 vs v0.5.10rc0

**Benchmark script**:
```bash
#!/bin/bash
ssh nvidia@192.168.1.76 << 'REMOTE'
set -euo pipefail

# Setup
HF_HOME=/data/hf_cache  # adjust to your HF cache path
MODEL=mistralai/Mixtral-8x7B-Instruct-v0.1

# Install both versions in separate venvs
python3 -m venv ~/bench/sglang-v059 && source ~/bench/sglang-v059/bin/activate
pip install -q sglang==0.5.9 && deactivate

python3 -m venv ~/bench/sglang-v0510 && source ~/bench/sglang-v0510/bin/activate
pip install -q sglang==0.5.10rc0 && deactivate

# Benchmark function
bench_sglang() {
  local venv=$1 tag=$2
  source ~/bench/${venv}/bin/activate

  # Start server
  python -m sglang.launch_server --model-path $MODEL --tp 1 --port 30000 &
  SERVER_PID=$!
  sleep 60  # wait for load

  # Run benchmark (1000 requests, varied lengths)
  python -m sglang.bench_serving \
    --backend sglang \
    --host 127.0.0.1 --port 30000 \
    --dataset-name random \
    --num-prompts 500 \
    --random-input-len 512 --random-output-len 256 \
    --request-rate 10 \
    --output-file /tmp/bench_${tag}.json

  kill $SERVER_PID
  deactivate
}

bench_sglang sglang-v059 v059
sleep 10
bench_sglang sglang-v0510 v0510

echo "=== Results ==="
python3 - << 'PY'
import json
for tag in ["v059", "v0510"]:
    d = json.load(open(f"/tmp/bench_{tag}.json"))
    print(f"\n{tag}:")
    print(f"  Throughput:  {d.get('output_throughput', 'N/A'):.1f} tok/s")
    print(f"  TTFT p50:    {d.get('ttft_percentiles', {}).get('p50', 'N/A'):.1f} ms")
    print(f"  TPOT p50:    {d.get('tpot_percentiles', {}).get('p50', 'N/A'):.2f} ms/tok")
PY
REMOTE
```

**Metrics**: throughput (tok/s), TTFT p50/p99 (ms), TPOT p50 (ms/tok), GPU peak memory (nvidia-smi)
**Pass criteria**: ≥5% throughput improvement OR ≥10% GPU memory reduction in v0.5.10

## Experiment 2 — vLLM v0.19.0 Regression Check

- Nodes: spark-01 (rank 0) + spark-02 (rank 1)
- Model: `meta-llama/Llama-3-70B-Instruct` (requires ~140GB → split across 2x128GB nodes)
- TP=2 via inter-node tensor parallelism (10GbE — expect NCCL bandwidth to be the bottleneck)

**Benchmark script**:
```bash
ssh nvidia@192.168.1.76 << 'REMOTE'

bench_version() {
  local version=$1
```

**Metrics**: TTFT p50/p99, TPOT p50, throughput, GPU memory (GB)
**Pass criteria**: No regression >5% on TTFT or TPOT p50; MTP logprob fix confirmed correct

## Experiment 3 — TRT-LLM Disaggregated KV Router (Chat API)

**Hypothesis**: TRT-LLM v1.3.0rc10 disaggregated KV-cache router now correctly routes prefill→decode for chat completions API (was broken before this fix). Measure baseline P/D latency split.

**Setup**:
- spark-01: prefill node (runs prompt processing)
- spark-02: decode node (runs token generation)
- Model: `meta-llama/Llama-3-8B-Instruct`
- Inter-node KV transfer over 10GbE (NIXL or NCCL)

**Script**:
```bash
# On controller, orchestrate both nodes
cat > /tmp/run_disagg.sh << 'EOF'
# Start prefill engine on spark-01
ssh nvidia@192.168.1.76 "
  cd /tmp && pip install -q tensorrt-llm==1.3.0rc10
  trtllm-serve meta-llama/Llama-3-8B-Instruct \
    --port 8001 --role prefill \
    --kv-cache-transfer-host 192.168.1.77 --kv-cache-transfer-port 9001 &
" &

# Start decode engine on spark-02
ssh nvidia@192.168.1.77 "
  trtllm-serve meta-llama/Llama-3-8B-Instruct \
    --port 8002 --role decode \
    --kv-cache-receiver-port 9001 &
" &

sleep 90  # wait for both to load

# Send chat API requests through the router (on spark-01)
python3 - << 'PY'
import openai, time, statistics

client = openai.OpenAI(base_url="http://192.168.1.76:8001/v1", api_key="none")
ttfts = []
for _ in range(50):
    start = time.perf_counter()
    resp = client.chat.completions.create(
        model="meta-llama/Llama-3-8B-Instruct",
        messages=[{"role": "user", "content": "Explain transformer attention in detail."}],
        max_tokens=200, stream=True
    )
    first = True
    for chunk in resp:
        if first and chunk.choices[0].delta.content:
            ttfts.append((time.perf_counter() - start) * 1000)
            first = False
print(f"TTFT p50: {statistics.median(ttfts):.1f} ms")
print(f"TTFT p99: {sorted(ttfts)[int(len(ttfts)*0.99)]:.1f} ms")
PY
EOF
bash /tmp/run_disagg.sh
```

**Metrics**: TTFT (prefill latency), TPOT (decode node), KV transfer size (GB), end-to-end throughput
**Pass criteria**: Requests complete correctly (no routing errors), TTFT < 500ms for 1K-token prompts

## Infrastructure Notes

- **NCCL over 10GbE**: Set `NCCL_SOCKET_IFNAME=eth0 NCCL_IB_DISABLE=1` — expect ~1.2 GB/s effective KV transfer bandwidth
- **HF model cache**: Recommend shared NFS or pre-downloading weights to both nodes at same path (`/data/hf_cache`)
- **GPU memory**: Each DGX Spark node has 128GB unified (NVLink-C2C) — models up to ~100B params fit single-node in BF16
- **Monitoring**: Check `nvidia-smi dmon -s u` for GPU utilization, `nethogs` for inter-node bandwidth during KV transfer experiments
